StockOps Hybrid Multi-Cloud Architecture

AWS Seoul Primary, AWS Tokyo Warm Standby, Azure/GCP Backup & Restore, On-Premise Sensimul edge, GitLab CI/CD를 한 화면에서 설명하기 위한 회의/발표용 설계 보드.

Primary: AWS Seoul Standby: AWS Tokyo Backup: Azure / GCP Edge: On-Premise CI/CD: GitLab

설계 요약

실제 구현은 AWS Seoul을 중심으로 안정화하고, AWS Tokyo는 Warm Standby로 리전 장애에 대비한다. Azure/GCP는 백업본과 배포 아티팩트를 보관해 AWS 전체 장애 시 복구 가능한 구조를 만든다.

핵심 판단

  • 전체 Active-Active는 최종 확장안으로 발표한다.
  • 재고/발주 쓰기 데이터는 Primary-Standby로 제한한다.
  • Sensimul은 온프레미스 현장 센서 시뮬레이터이며, 센서값은 Sensimul -> AWS IoT Core -> API 흐름이다.
  • 센서 스트림은 기본적으로 DB 저장 대상이 아니며, RDS는 재고/출고/발주 등 업무 데이터를 저장한다.
  • 8개 센서가 초당 1회씩 publish하면 월 약 2,073만 메시지이며, 단순 publish 기준 약 $20 전후로 추정한다.
  • AI와 정적 웹은 멀티클라우드 standby 또는 active-active 후보로 둔다.
  • Terraform은 인프라, GitLab CI/CD는 앱 릴리즈를 담당한다.

회의 결정 포인트

  • Tokyo standby를 실제 구축할지, 시뮬레이션으로 둘지
  • Azure/GCP 백업을 둘 다 구현할지
  • MQTT Broker를 AWS IoT Core로 대체할지, Mosquitto fallback을 남길지
  • AI Module을 GCP Cloud Run에도 둘지
  • deploy job을 자동화할지 수동 승인으로 둘지
